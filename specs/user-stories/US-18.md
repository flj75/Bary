# US-18 — Ouvrir un lien partagé et afficher le résultat

**Récit**
En tant que destinataire d'un lien Bary, je veux qu'en ouvrant le lien dans mon navigateur je voie directement l'écran 4 avec le résultat pré-calculé, afin d'accéder au point de rencontre sans refaire le calcul.

**Priorité** : Should
**Statut** : Livré

---

### Critères d'acceptation

**Scénario 1 — Ouverture du lien partagé**
- **Given** un destinataire reçoit le lien `bary.app/result?s=[station-id]&g=[prenom|station-id,...]`
- **When** il ouvre le lien dans son navigateur
- **Then** il voit directement l'écran 4 avec la station optimale, les participants et leurs temps de trajet

**Scénario 2 — Lien malformé ou station introuvable**
- **Given** le lien contient un stationId invalide ou manquant
- **When** la page se charge
- **Then** l'utilisateur est redirigé vers l'Écran 1 avec un message "Lien invalide"

---

### Dépendances
- `US-10` — l'émetteur génère le lien
- `BUG-01` corrigé — séparateur `:` → `|` dans `buildShareUrl`
- `BUG-03` corrigé — validation à la saisie du prénom (caractères `, | & = + # ?` interdits)

### Edge cases identifiés
- Lien malformé ou station introuvable → redirection Écran 1 + toast "Lien invalide"
- StationId IDFM contenant `:` → résolu par BUG-01 (séparateur `|`)
- Nom contenant `|` → résolu par double-encodage côté émetteur (`%7C` → `%257C`)
- Nom contenant `,` → bloqué à la saisie (BUG-03 corrigé)
- Participant avec station introuvable dans le dataset → redirection "Lien invalide"

### Notes techniques / Contraintes connues
- Format URL : `bary.app/result?s=[station-id]&g=[prenom|station-id,prenom|station-id,...]`
- Séparateur `|` entre prénom et stationId (BUG-01 corrigé)
- Les noms contenant `|` sont double-encodés côté émetteur (`%7C` → `%257C`) et double-décodés côté récepteur — transparent pour les prénoms normaux
- Les caractères `, | & = + # ?` sont interdits à la saisie (BUG-03 corrigé) — validation inline dans la modale "+ Nouvelle personne"
- Spinner immédiat côté destinataire (`urlLoading` initialisé à `!result`) — pas de flash blanc
- La page reconstitue le résultat via `findMeetingPoint` — les temps de trajet ne sont pas encodés dans l'URL, ils sont recalculés à l'ouverture
- Pas de backend requis : tout est reconstruit côté client uniquement

### Lien maquette
- Écran 4 — Résultat (vue destinataire)
