# US-17 — Gestion d'un calcul impossible

**Récit**
En tant qu'utilisateur, si le calcul ne trouve pas de trajet possible entre deux stations, je veux voir un message d'erreur explicite plutôt qu'un résultat vide ou un plantage, afin de comprendre ce qui se passe.

**Priorité** : Should
**Statut** : Livré

---

### Critères d'acceptation

**Scénario 1 — Calcul aboutissant à un résultat incohérent (filet de sécurité post-fallback)**
- **Given** le groupe contient des participants sur des stations très périphériques ou orphelines dans le dataset, et le fallback "candidates = allStations" est activé (aucune station accessible en ≤ 1 correspondance pour tous)
- **When** le minimax s'exécute sur l'ensemble du réseau (~950 stations)
- **Then** un résultat est retourné normalement — ce scénario confirme que le fallback défini dans `specs/04-algorithm.md` (Étape 2b) absorbe les cas de réseau déconnecté et qu'aucune erreur n'est affichée à l'utilisateur dans ce cas

**Scénario 2 — Dataset vide ou candidates vides après fallback (cas résiduel)**
- **Given** le dataset est chargé mais ne contient aucune station (fichier vide ou malformé), ou toutes les stations sont filtrées par une erreur logique
- **When** le minimax est lancé avec une liste candidates vide
- **Then** le calcul est interrompu, aucun résultat n'est affiché, et un message s'affiche : "Le calcul n'a pas pu aboutir. Vérifiez les stations sélectionnées ou réessayez." avec un bouton "Modifier le groupe" (retour écran 2)

**Scénario 3 — Tous les participants à la même station**
- **Given** le groupe contient uniquement des participants partageant la même station de départ
- **When** le calcul s'exécute
- **Then** cette station elle-même est retournée comme résultat (temps de trajet = 0 min pour tous), les barres de progression s'affichent toutes à 100% (égalité parfaite, cas documenté dans US-08) — ce n'est pas une erreur

---

### Dépendances
- `US-06` — le calcul est déclenché depuis l'écran Paramètres
- `US-07` — l'écran Résultat suppose qu'un résultat valide existe ; US-17 définit ce qui se passe quand ce n'est pas le cas
- `specs/04-algorithm.md` (Étape 2b) — le fallback vers `allStations` rend le Scénario 2 quasi-impossible en conditions normales ; US-17 est un filet de sécurité pour les cas de corruption de données ou de bug logique

### Edge cases identifiés
- Station orpheline dans le dataset (présente dans `stations.json` mais sans lignes associées) : `getReachableStations` retourne un ensemble vide pour ce participant, l'intersection est vide, le fallback s'active — résultat retourné normalement
- Dataset avec une seule station : le minimax retourne cette station, temps = 0 pour tous les participants qui y sont, et haversine pour les autres — pas d'erreur, résultat potentiellement absurde mais non bloquant

### Notes techniques / Contraintes connues
- Le fallback défini dans `specs/04-algorithm.md` (Étape 2b) garantit qu'un résultat est toujours retourné tant que le dataset contient au moins une station : `findOptimalStation` ne peut retourner `null` que si `candidates` est vide après fallback — seul cas réel de Scénario 2
- La fonction `findOptimalStation` initialise `best` à `null` : le code appelant doit vérifier ce cas et déclencher l'affichage d'erreur si `best === null`
- Délai artificiel de 1 200 ms (Écran 3 → Écran 4) : en cas d'erreur détectée pendant ce délai, l'animation de chargement doit s'interrompre et l'erreur s'afficher à la place de la transition
