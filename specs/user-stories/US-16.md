# US-16 — Gestion d'une station invalide

**Récit**
En tant qu'utilisateur, si la station que je saisis pour un participant n'existe pas dans le réseau IDFM, je veux voir un message d'erreur clair, afin de corriger ma saisie.

**Priorité** : Must
**Statut** : Livré

---

### Critères d'acceptation

**Scénario 1 — Échec de chargement du dataset (cas principal)**
- **Given** je suis dans la modale "Nouvelle personne" et le dataset IDFM n'a pas pu se charger (erreur réseau, fichier corrompu)
- **When** le champ station s'affiche
- **Then** le champ est désactivé et un message explicite s'affiche : "Impossible de charger les stations. Vérifiez votre connexion et réessayez." avec un bouton "Réessayer"

**Scénario 2 — Station absente du dataset après sélection (filet de sécurité)**
- **Given** un participant a été ajouté avec une station valide, mais une incohérence dans le dataset fait que son `stationId` ne se résout plus à l'exécution du calcul
- **When** le calcul minimax est lancé
- **Then** le calcul est interrompu avant d'aboutir à un résultat silencieusement erroné, et un message s'affiche : "Une station du groupe est introuvable dans le réseau. Veuillez revérifier les stations des participants." avec un retour vers l'écran 2

**Scénario 3 — Saisie libre contournée (défense en profondeur)**
- **Given** l'autocomplete est fonctionnel mais l'utilisateur parvient à soumettre le formulaire avec une valeur qui n'est pas dans la liste (ex: manipulation du DOM, bug d'interface)
- **When** la modale est validée
- **Then** la validation côté client rejette la soumission et affiche une erreur inline : "Veuillez sélectionner une station dans la liste."

---

### Dépendances
- `US-03` — cette US est le filet de sécurité de l'autocomplete défini dans US-03 : en fonctionnement normal, l'autocomplete empêche toute saisie invalide (saisie libre non acceptée). US-16 couvre les cas résiduels où cette protection échoue (erreur de chargement du dataset, bug d'interface, incohérence de données).

### Edge cases identifiés
- Dataset partiellement chargé (truncature réseau) : certaines stations peuvent manquer sans que l'erreur soit détectée à l'ouverture de la modale — le Scénario 2 couvre ce cas au moment du calcul
- Rechargement partiel du dataset après le Scénario 1 : si le retry réussit mais que le dataset est différent de celui utilisé pour ajouter d'autres participants, un avertissement peut apparaître au calcul (Scénario 2)

### Notes techniques / Contraintes connues
- Le chargement du dataset (`/data/stations.json`) est asynchrone au premier rendu client : l'état de chargement (loading / error / success) doit être géré dans le composant `StationAutocomplete`
- La validation finale de l'existence du `stationId` doit être réalisée dans `buildCandidates` (voir `specs/04-algorithm.md`, Étape 2b) avant le lancement du minimax
- Aucun cas utilisateur courant attendu en production : le dataset est embarqué dans `public/data/` et servi depuis le même domaine — les erreurs réseau sont rares mais non impossibles (hors-ligne, CDN)
